# LineArt Metadata

English · [日本語](../../ja/image/lineart-metadata.md)

LineArt Metadata is a thin shared layer for adding roles, connections, containment, and validation expectations to each shape in LineArt JSON. It keeps rendering simple while giving later geometry validation and LLM repair feedback a common set of information to read.

## When To Use This Document

Use this when creating diagrams, flowcharts, UI mockups, logo drafts, or other LineArt JSON drawings.

The minimal prompt can be:

```text
Add metadata to this LineArt JSON.
```

With that prompt, the assistant adds fields such as `role`, `tags`, `container_id`, `connects_to`, and `validation` where useful. The user can inspect the generated JSON and, later, the validation report produced by `lineart validate`.

## Basic Policy

The first version should not overbuild a constraint DSL. Define thin metadata that every shape can carry, then let geometry validation read that information.

Fields to decide early:

- `id`
- `role`
- `name`
- `label`
- `tags`
- `description`
- `locked`
- `visible`

Thin fields for geometry validation:

- `container_id`
- `connects_to`
- `allow_overlap`
- `expected_position`
- `validation`

Defer these:

- Complex constraint expressions
- Automatic layout constraints
- Boolean-geometry-based conditions
- Priority-based repair rules

## Recommended Fields

| Field | Type | Purpose |
|-------|------|---------|
| `id` | string | Uniquely references the shape. |
| `role` | string | Describes the shape role, such as `container`, `node`, `connector`, `label`, or `glyph`. |
| `name` | string | Short human-readable name. |
| `label` | string | Text shown in the drawing. |
| `tags` | string[] | Optional classification, such as `annotation`, `primary`, or `debug`. |
| `description` | string | Notes about generation intent or validation. |
| `locked` | boolean | Marks a shape that LLM repair should not move. |
| `visible` | boolean | Marks whether the shape should be rendered. |
| `container_id` | string | The `id` of the container this shape should stay inside. |
| `connects_to` | object | Targets connected by an arrow or line. |
| `allow_overlap` | boolean | Whether overlap with other shapes is allowed. |
| `expected_position` | object | Position expectations such as inside, near, or aligned. |
| `validation` | object | Validation expectations and severity. |

## Initial role Values

| role | Meaning |
|------|---------|
| `container` | Panel, frame, or closed region |
| `node` | Flowchart or architecture element |
| `connector` | Arrow or connecting line |
| `label` | Text describing a target |
| `annotation` | Helper line, note, or emphasis |
| `glyph` | FontVector-derived text outline |
| `decorative` | Decoration that may be excluded from validation |
| `background` | Background or base layer |

## Label Example

```json
{
  "id": "label_01",
  "type": "text",
  "role": "label",
  "tags": ["annotation"],
  "container_id": "panel_01",
  "allow_overlap": false,
  "validation": {
    "expected": {
      "inside": "panel_01",
      "near": {
        "target": "icon_03",
        "max_distance": 24
      }
    },
    "severity": "error"
  }
}
```

## Connector Example

```json
{
  "id": "arrow_01",
  "type": "path",
  "role": "connector",
  "connects_to": {
    "from": "node_a",
    "to": "node_b"
  },
  "validation": {
    "expected": {
      "touches": ["node_a", "node_b"],
      "no_intersections": true
    },
    "severity": "error"
  }
}
```

## Relationship To FontVector

Text outlines generated by FontVector should use `role: "glyph"`. A group around the text can use `role: "label"` or `role: "decorative"` depending on whether it should participate in validation.

```json
{
  "id": "glyph_title_G",
  "type": "path",
  "role": "glyph",
  "tags": ["fontvector", "title"],
  "source_text": "GraphAssist",
  "container_id": "title_group"
}
```

## Update Policy

This page should evolve alongside the LineArt implementation. Before adding a new validation feature, first check whether existing metadata fields can express it. Add a new field only when the same need appears across multiple shape types.

Change priority:

1. Express the need with existing fields.
2. Add a small value under `validation.expected`.
3. Promote it to a shared field when multiple shape types need it.
4. Keep complex constraint expressions in a later constraint DSL.
